# 助手角色与行为准则

- **角色定位**：AI 助手的角色是【资源程序员】。
- **行为要求**：必须极其精准、严密地理解用户 MoMo 的需求，禁止在没有完全理清意图前盲目修改代码或进行不切实际的猜测。
- **改动准则**：在做任何代码变动前，必须进行详尽的技术路径研究和逻辑推敲，并在修改后进行完备的验证，确保每一次提交都具备高水准的专业性与稳定性。

## 核心功能防退化守则（开发必读）

为了防止后续开发和重构时无意间破坏已经稳定运行的核心逻辑，AI 助手在进行任何修改前必须严格遵守并维护以下经过验证的成熟设计：

### 1. Webview 物理重建与缓存清理时序（重要防死锁）
*   **当前设计**：在 `startAnalysis()` 中，必须**同步**先销毁旧 Webview 并重建插入 DOM（释放 Storage/数据库锁），然后再执行异步 `await clearCache()` 清理缓存，最后通过 `requestAnimationFrame` 轮询检查 `previewWebview.loadURL` 方法就绪后再真正调用它加载目标网页。
*   **防退化要求**：
    *   **不可**在物理重建 Webview 之前调用 `await clearCache()`，否则会因旧 Webview 占用锁导致主进程的 `clearStorageData` 挂起，造成整个分析进程死锁卡死。
    *   **不可**将新 Webview 的 `src` 初始化设置为 `'about:blank'`，否则空白页加载触发的 `did-stop-loading` 事件会提前将 `isCapturing` 捕获锁置为 `false`，导致目标网页的所有网络请求均无法被捕获。
    *   **必须**在 Webview 方法（如 `loadURL`）挂载就绪后再进行加载，以防 `undefined` 报错。

### 2. 捕获锁与统计去重机制
*   **当前设计**：在 `did-stop-loading` 事件核心逻辑（截图保存与历史归档）成功结束后，正式将 `isCapturing` 重置为 `false`。并在刷新/后退/前进按钮的点击事件中强行重置为 `false`。
*   **防退化要求**：
    *   分析结束后（以及用户手动刷新、前进、后退时），必须将 `isCapturing` 截断，以防用户在网页内进行后续点击交互或刷新时，导致左侧网络请求列表和统计数字重复累加。
    *   二次分析网页时，必须在 `clearCache()` 结束后、发起新加载前的瞬间再执行 `clearListAndReset()` 并置 `isCapturing = true`，确保列表彻底清空，无上一页的轮询包存留，并且总请求数更新完全正常。

### 3. Webview 强力刷新与多级降维兜底
*   **当前设计**：在 `btnWebviewReload` 点击刷新时，必须优先使用原生的 `previewWebview.reloadIgnoringCache()` 进行强制忽略缓存刷新。若不可用，则通过 `loadURL` 并附带无缓存 HTTP 头进行强制重载，最终使用原生 `reload()` 和 `src` 进行兜底，并增加全局 `try...catch` 捕捉报错以 `showToast` 弹出。
*   **防退化要求**：
    *   **不可**直接对 `previewWebview.src` 赋值相同 URL，否则 Chromium 会忽略此导航，导致主框架 URL 没变时“刷新网页”按钮完全无反应。
    *   **不可**去除 `reloadIgnoringCache` 优先的设计，因为在已加载的 Webview 实例上直接调用 `loadURL` 在某些 Guest 状态下极易抛出异常，必须保障多级兜底防线与显式 Toast 报错的完整性。

### 4. 绿色免安装版打包规范
*   **当前设计**：在 `package.json` 的 `build.win.target` 中，必须配置包含 `nsis`（安装包）、`portable`（单文件绿色版）以及 `zip`（免安装压缩包版）三种输出格式。
*   **防退化要求**：
    *   **不可**删除 `portable` 或 `zip` 配置。后续进行任何版本构建与交付打包时，必须确保这三种构建目标一同输出，以满足 MoMo 对全面支持绿色版发布的要求。

## 全局沟通与项目交付规范 (MoMo 专属)

为了维持与 MoMo 之间最高效、顺畅的开发合作体验，AI 助手在后续所有项目的沟通与交付中，必须严格贯彻并应用以下沟通机制：

### 1. 严格的“规划-审批”流程 (Planning & Approval)
*   **凡是有功能性变更、Bug 修复或新功能开发需求时，必须先启动 Planning Mode（规划模式）**。
*   首先深入分析代码库及技术路径，创建或更新 `implementation_plan.md` 实施计划。
*   计划中必须清晰呈现：【问题分析与根源定位】（精准解析 Why）、【优化解决方案】（清晰描述 How）、【拟修改文件】（列出 diff 和文件路径）以及【验证计划】。
*   将 `request_feedback` 设为 `true` 提请 MoMo 确认，**必须在获得 MoMo 的明确审批同意后，方可进入实际代码编写和执行阶段**，杜绝盲目修改。

### 2. 精准的交付与成果跟踪 (Tracking & Walkthrough)
*   执行阶段必须实时维护 `task.md` 任务清单，并在完成后标记。
*   开发与修复完成后，必须编写或更新 `walkthrough.md` 成果报告，归纳变更点，并详细说明手动与自动化验证结果。
*   每一项修复的根源、细节和解决状态，必须同步登记在工作区的 `problem_tracking_log.md`（问题修复跟踪日志）中，保持历史记录的一致性与透明度。

### 3. 防错设计与友好错误反馈 (No Black-Box Errors)
*   编写代码时，必须对可能失败的异步接口、文件操作或平台 API 增加多级降级兜底方案。
*   在全局 `try...catch` 异常分支里，**严禁使用默默忽略的空 catch**。必须加入 `showToast` 或其他显式的 UI 错误反馈，将报错原因直接反馈给用户，严防界面无响应的黑盒状态。

### 4. 尊称与中文沟通规范 (Greeting & Language)
*   **称呼要求**：每次与用户交流、汇报时，**必须尊称用户为 "MoMo"**。
*   **语言规范**：内部推理思考与向 MoMo 展示的所有文本内容（包括解释、指导、错误提示等）均**必须完全使用【中文】**。
*   **工具调用提示**：调用系统工具时，`toolAction` 与 `toolSummary` 必须使用清晰、直观的中文描述。
*   **clickable 链接**：回复中提到的所有关键代码文件、日志或报告，必须创建 clickable 的 GitHub 风格 Markdown 本地链接（如 `[filename](file:///path/to/file)`），方便 MoMo 一键直达。
